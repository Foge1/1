# ADR-0007: Feature modularization strategy (одна фича = один модуль)

## Status
Accepted

## Context
Проект развивается от монолита к feature-модульной структуре. Без явной стратегии новые фичи начинают делить data-слой и напрямую импортировать чужие реализации, что увеличивает связность и стоимость изменений.

## Decision
1. Базовая единица масштабирования — один feature-модуль на одну продуктовую фичу (`:feature-<name>`).
2. Внутри feature-модуля допускается вертикальный срез: `data + domain + presentation + ui`.
3. Межфичевое общение — только через API/интерфейсы (domain/api контракты), без импортов чужого `data`.
4. Feature-owner владеет persistence-слоем фичи (`Entity/Dao/DB/migrations`).

## Consequences
- Ясные ownership-границы и изоляция изменений по фичам.
- Проще выделять команды/ответственность по модулям.
- Увеличивается количество интеграционных контрактов (их нужно поддерживать явно).
